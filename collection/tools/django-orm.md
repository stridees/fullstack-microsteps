# Django ORM — модели вместо ручного SQL

**Боль.** На голом `sqlite3` таблицу заводишь руками (`CREATE TABLE`), данные
кладёшь и достаёшь запросами (`INSERT`, `SELECT`), возишься с соединением,
курсором, `commit` и следишь за SQL-инъекциями. Много рутины и легко ошибиться.

**Когда использую.** Когда данные надо хранить в базе и работать с ними из Python:
описать таблицу, добавлять, искать, считать записи.

**Команды-минимум.** ORM (*Object-Relational Mapping*) связывает класс с таблицей,
объект — со строкой, атрибут — с колонкой:
- модель — класс: `class Quote(models.Model): text = models.TextField()`.
- `Quote.objects.create(...)` — добавить (вместо `INSERT`).
- `Quote.objects.all()` / `.filter(author='…')` / `.get(id=1)` — выбрать.
- `Quote.objects.count()` / `.first()` — посчитать, взять одну.
- `python manage.py shell` — интерактивно поиграть с моделями.

**Как надо.**
- Лучшие практики: для коротких строк — `CharField(max_length=…)`, для длинного
  текста — `TextField`; добавляй `__str__`, чтобы объект читался; фильтруй данные в
  базе через `filter()`, а не перебором в Python.
- Личные находки: *— фиксируй всё, что относится к личному опыту.*

**Как НЕ надо.**
- Антипаттерны: тащить `.all()` в память и фильтровать циклом вместо `filter()`;
  писать сырой SQL там, где хватает ORM.
- Мои грабли: объект печатался как `Quote object (1)`, пока не добавил `__str__`;
  менял поле модели и забывал миграцию. Критично: изменил поля модели — нужны
  `makemigrations` + `migrate`, иначе база отстанет от кода.
