# Миграции — версии структуры базы

**Боль.** Поменял модель в коде — а таблицы в базе ещё нет или она устарела.
Править структуру руками (`ALTER TABLE`) — больно, и код с базой быстро
расходятся: «у меня работает, у тебя падает».

**Когда использую.** Каждый раз, когда добавил или изменил модель либо её поле.

**Команды-минимум.**
- `python manage.py makemigrations [app]` — *make migrations*, записать план
  изменения в файл-миграцию (это ещё не база).
- `python manage.py migrate` — *migrate*, применить план к базе.
- `python manage.py showmigrations` — что применено, что нет.

**Как надо.**
- Лучшие практики: рабочий цикл «поменял модель → `makemigrations` → `migrate`»;
  файлы из `migrations/` коммить в git — это версии схемы, как коммиты для кода;
  загляни в созданный файл, прежде чем применять.
- Личные находки: *— фиксируй всё, что относится к личному опыту.*

**Как НЕ надо.**
- Антипаттерны: править структуру базы руками в обход миграций; не коммитить файлы
  миграций (тогда схему не воспроизвести).
- Мои грабли: поменял поле и забыл `makemigrations` — база отстала от модели;
  пытался мигрировать после правки `__str__`, а Django сказал «No changes
  detected» (это поведение, не схема). Критично: миграции — часть кода, всегда в
  git.
