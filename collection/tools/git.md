# Git — машина времени для кода

**Боль.** Поменял строчку — всё сломалось, а как было — непонятно.
Плодить копии `main_финал.py`, `main_финал2.py` — болото без истории и без
возможности откатиться. Git хранит снимки проекта (коммиты), и к любому можно
вернуться. (Само слово *git* — не сокращение, а название от создателя Linux Линуса
Торвальдса.)

**Когда использую.** В начале любого проекта (`git init`) и дальше — каждый
раз, когда сделал законченный кусок работы и хочу его зафиксировать.

**Команды-минимум.**
- `git init` — *initialize*, завести репозиторий в текущей папке.
- `git status` — *status*, что git видит сейчас.
- `git add .` — *add*, пометить файлы в будущий снимок (`.` — все в папке).
- `git diff --staged --stat` — короткий список того, что войдёт в коммит.
- `git commit -m "..."` — *commit*, снимок с подписью (`-m` — *message*).
- `git push` — *push*, отправить коммиты на GitHub.

**Как надо.**
- Лучшие практики: коммить маленькими законченными кусками с понятной подписью;
  перед каждым коммитом смотри `git status` и `git diff --staged --stat`, чтобы в
  снимок не попало лишнее.
- Личные находки: *— фиксируй всё, что относится к личному опыту.*

**Как НЕ надо.**
- Антипаттерны: плодить копии `main_финал.py`, `main_финал2.py` вместо истории;
  делать `git add .`, не глядя, что войдёт в снимок.
- Мои грабли: чуть не сделал первый коммит до `.gitignore` — «сфотографировал» бы
  тысячи файлов из `.venv` (сначала `.gitignore`, потом первый коммит). Критично:
  папку `.git` руками не трогай — удалишь, потеряешь всю историю.
