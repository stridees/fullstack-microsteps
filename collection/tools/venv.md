# venv — виртуальное окружение

**Боль.** Два проекта дерутся за разную версию Django; глобальная
установка сталкивает их лбами, мусорит в системном Python и требует прав админа.
venv (*virtual environment* — «виртуальное окружение») даёт каждому проекту свой,
отдельный от системы Python с библиотеками. Удалил папку `.venv` — стало чисто.

**Когда использую.** Первым делом в новом проекте — до установки любой
библиотеки.

**Команды-минимум.**
- `python -m venv .venv` — создать окружение в папке `.venv` (`-m` — *module*).
- Активировать:
  - Windows / PowerShell: `.venv\Scripts\Activate.ps1`
  - macOS/Linux: `source .venv/bin/activate`
- `deactivate` — выйти из окружения.

**Как надо.**
- Лучшие практики: заводи venv до первой установки; держи его в `.venv` внутри
  проекта; проверяй, что Python берётся из окружения — `Get-Command python`
  (macOS/Linux: `which python`) ведёт внутрь `.venv`.
- Личные находки: *— фиксируй всё, что относится к личному опыту.*

**Как НЕ надо.**
- Антипаттерны: ставить пакеты в системный Python «на весь компьютер»; коммитить
  папку `.venv` в git.
- Мои грабли: открыл новый терминал, забыл активировать — команды летят мимо
  окружения (признак: нет приставки `(.venv)`). PowerShell не пускал `Activate.ps1`
  («running scripts is disabled») — лечится `Set-ExecutionPolicy -Scope Process
  -ExecutionPolicy Bypass` на текущее окно. Критично: `.venv` — в `.gitignore`,
  иначе раздуешь репозиторий тысячами файлов.
