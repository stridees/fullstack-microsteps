# Django — веб-фреймворк

**Боль.** На голом Python приём запроса, разбор адреса, поиск
обработчика и сборка HTML — это сотни строк низкоуровневого кода. Django (название,
не аббревиатура) берёт рутину на себя: маршруты, шаблоны, базу данных, админку. Нам
остаётся логика тренажёра.

**Когда использую.** Когда строю серверную часть сайта: страницы, данные,
маршруты.

**Команды-минимум.**
- `django-admin startproject trainer_project .` — *start project*, создать
  структуру. Точка `.` — «в текущей папке»; без неё будет лишняя папка-матрёшка.
- `python manage.py runserver` — *run server*, поднять сервер разработки
  (`manage.py` — «пульт» проекта).
- `python manage.py startapp trainer_app` — *start app*, создать приложение.
- `python manage.py check` — проверить настройки, приложения и маршруты.

**Цепочка запроса (запомни).**
`браузер → urls.py (какой адрес?) → views.py (что ответить?) → шаблон (как
выглядит) → браузер`. Своя страница собирается из трёх кусочков: view-функция,
шаблон `templates/<app>/index.html` и маршрут в `urls.py` приложения, подключённый
в главный `urls.py` через `include()`.

**Как надо.**
- Лучшие практики: чаще зови `python manage.py check` — он ловит ошибки в настройках
  и маршрутах раньше, чем ты откроешь страницу; регистрируй приложение через
  `trainer_app.apps.TrainerAppConfig`; шаблоны клади в `templates/<app>/`.
- Личные находки: *— фиксируй всё, что относится к личному опыту.*

**Как НЕ надо.**
- Антипаттерны: складывать логику сайта в папку настроек `trainer_project` вместо
  приложений; запускать проект, игнорируя предупреждения `check`.
- Мои грабли: забыл про `INSTALLED_APPS` — Django «не видел» приложение; положил
  шаблон в `templates/index.html` вместо `templates/trainer_app/index.html`; забыл
  точку в `startproject … .` (лишняя папка-матрёшка). Критично: `SECRET_KEY` из
  `settings.py` в настоящем сервисе нельзя хранить в git — его выносят в `.env`.
