# pip — установщик пакетов Python

**Боль.** Чистый Python не умеет «в веб» и многого другого. Нужные
библиотеки (например, Django) кто-то должен скачать и положить в проект. pip
(*Package Installer for Python* — «установщик пакетов для Python») берёт их из
общего каталога PyPI (*Python Package Index*, pypi.org).

**Когда использую.** Когда проекту нужна внешняя библиотека — и обязательно
внутри активированного venv, чтобы она легла в окружение проекта, а не в систему.

**Команды-минимум.**
- `python -m pip install "Django>=6.0,<6.1"` — *install*, поставить пакет. Запуск
  через `python -m` берёт тот pip, что внутри `.venv`.
- `python -m pip list` — что уже стоит в окружении.

**Как надо.**
- Лучшие практики: запускай pip как `python -m pip`, а не просто `pip` — сработает
  тот, что внутри активного окружения; фиксируй версии (`"Django>=6.0,<6.1"`),
  чтобы команды и выводы были предсказуемы.
- Личные находки: *— фиксируй всё, что относится к личному опыту.*

**Как НЕ надо.**
- Антипаттерны: ставить пакеты в системный Python на весь компьютер; полагаться на
  библиотеку, случайно оставшуюся от другого проекта.
- Мои грабли: ставил пакет, забыв активировать venv (нет приставки `(.venv)`) — он
  улетал в систему. Критично: ставь только при активном окружении.
