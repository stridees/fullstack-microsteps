# Шаг 1. Первый Django-проект с нуля

> **К чему придём в этом шаге.** Начнём с пустой папки и закончим работающим
> сайтом, который честно сообщает посетителю: «Сайт находится в разработке,
> зайдите к нам позже». Это уже практическая польза — посетитель видит, что сайт
> живой. Всё это будет лежать в виртуальном окружении, под контролем git и
> опубликовано на GitHub.
>
> **Что НЕ делаем в этом шаге.** Пока сайт ничего не *помнит*: нет ни базы
> данных, ни моделей. Это намеренно — к базе данных мы подойдём в конце шага,
> наткнувшись на боль, которую без неё не вылечить.

> **Про команды и операционные системы.**
> Основные команды даны для Windows (PowerShell), потому что это самая частая
> система у новичков. Где команда для macOS/Linux отличается, есть отдельная
> вставка с пометкой «macOS/Linux». В блоках кода строка, начинающаяся с `#`, — это
> комментарий, его вводить не надо. У каждой команды есть расшифровка: из каких
> английских слов она составлена — так её проще запомнить, а не зубрить.

## 1.0. Что мы вообще строим

### Большая цель раздела

К концу этого раздела курса (это несколько крупных шагов) у нас будет backend
веб-сервиса для совместного обучения. Backend — это часть сайта, которая работает
на сервере: хранит данные, проверяет правила и отвечает браузеру. Идея сервиса
такая:

- сервис помогает людям отслеживать прогресс в изучении какой-либо темы
  (например, Python);
- у каждого человека есть своя страница прогресса: какие темы и когда он изучил,
  какие задания и как уже решил, что вызвало трудности;
- по этим данным система умеет подбирать напарников с похожим уровнем и
  интересами, чтобы они могли тренироваться вместе.

То есть мы строим не «сайт с задачками», а платформу, которая знает про каждого
пользователя: где он сейчас, что ему даётся тяжело и с кем его полезно свести.
Именно поэтому дальше нам понадобятся база данных (хранить прогресс), модели
(описать «человека», «тему», «задание», «попытку»), связи между ними и логика
подбора напарников.

### Учебный проект — «Python-тренажёр»

Чтобы прийти к этой большой цели маленькими победами, мы ведём один сквозной
проект — «Python-тренажёр». Это первый кирпич будущей платформы: сайт, где
пользователь решает задачки по Python (а позже — где копится его прогресс,
который и станет основой для подбора напарников).

Большое дело пугает. Поэтому разбиваем его на микрошаги: на каждом шаге —
маленькая победа, которую видно глазами. Сегодняшняя победа маленькая, но важная:
свой сайт, который открывается в браузере и лежит на GitHub.

Прежде чем учить базу данных, маршруты, модели и логику подбора, нам нужен
«скелет», на который всё это можно вешать. Этим скелетом и займёмся.

## Коллекции перед шагом

В этом курсе у тебя постоянно растут две коллекции. **Коллекция инструментов** —
чем ты умеешь пользоваться (и всё, что ты про каждый инструмент знаешь: грабли,
технику безопасности, приёмы). **Коллекция проектов** — что ты собрал этими
инструментами и можешь показать. Обе живут в папке
[`collection/`](../collection/README.md) — туда мы и будем заводить карточки.

Перед первым шагом Коллекция инструментов небольшая:
- базовый Python: переменные, условия, циклы, функции, ООП, импорт библиотек;
- умение открыть браузер, скачать программу и установить её обычным способом;
- умение читать инструкции и аккуратно выполнять команды по порядку.

Коллекция проектов пока пустая. Это нормально: первым предметом в ней станет
«Python-тренажёр», который мы начнём в этом шаге.

## 1.1. Проверяем Python и открываем терминал

Мы собираемся ставить Django и запускать команды. Но мы пока даже не убедились,
что Python установлен и что мы умеем «разговаривать» с компьютером текстом, а не
мышкой. А ещё: Django 6 работает не на любом Python — на старом он просто не
запустится.

### Решение

**Терминал** — окно, где компьютеру отдают команды текстом. Это основной
инструмент разработчика: быстро, точно и воспроизводимо (команду можно
скопировать, повторить, дать коллеге).

Открой терминал:
- **Windows:** нажми `Win`, набери `PowerShell`, открой Windows PowerShell.
  (Или в VS Code: меню `Terminal → New Terminal`.)
- **macOS:** `Cmd+Space`, набери `Terminal`.
- **Linux:** `Ctrl+Alt+T`.

Проверь версию Python (флаг `--version` — «показать версию»):
```powershell
python --version
```
Должно напечатать что-то вроде `Python 3.13.1`. Для Django 6 нужен Python 3.12
или новее.

> На macOS/Linux часто нужно писать `python3` вместо `python`:
> ```bash
> python3 --version
> ```
> На Windows бывает удобнее «лаунчер» `py`:
> ```powershell
> py --version
> ```

Если Python не нашёлся или версия меньше 3.12 — скачай свежий с
[python.org/downloads](https://www.python.org/downloads/). На Windows при
установке обязательно поставь галочку **«Add Python to PATH»**. PATH — список
папок, где система ищет команды вроде `python`.

В терминале пригодятся базовые команды передвижения по папкам:

| Действие | Команда |
|---|---|
| Где я сейчас? | `pwd` |
| Что в этой папке? | `ls` |
| Зайти в папку `projects` | `cd projects` |
| Подняться на уровень вверх | `cd ..` |
| Создать папку `projects` | `mkdir projects` |

Названия команд — это сокращения английских слов:
- `pwd` — *print working directory*, «показать текущую папку»;
- `ls` — *list*, «перечислить содержимое»;
- `cd` — *change directory*, «сменить папку» (а `..` означает «папка на уровень выше»);
- `mkdir` — *make directory*, «создать папку».

Эти команды — 90% работы в терминале на старте. Остальное добавим по мере
появления боли.

## 1.2. Папка проекта

Код должен где-то жить. Если бросать файлы на рабочий стол вперемешку со
скриншотами и мемами — через неделю мы сами не найдём свой проект, а инструменты
вроде git и Django запутаются, где границы проекта.

### Решение

Одна отдельная папка под весь проект. Всё, что относится к «Python-тренажёру»,
живёт внутри неё — и ничего постороннего.

Сначала перейди туда, где хранишь учебные проекты (например, в домашнюю папку),
потом создай папку проекта и зайди в неё:
```powershell
# перейти в домашнюю папку (пример — подставь свой путь при желании)
cd ~

# создать папку проекта
mkdir python-trainer

# войти в неё
cd python-trainer
```
Тильда `~` — короткое обозначение твоей домашней папки, чтобы не писать полный
путь целиком.

Проверим, где мы оказались:
```powershell
pwd
```
Путь должен заканчиваться на `...\python-trainer`. Дальше все команды шага
выполняем, находясь внутри этой папки.

## 1.3. Git: машина времени для кода

Ты пишешь код, всё работает. Меняешь одну строчку — всё ломается. А как было?
Непонятно. Начинаешь плодить копии `main_финал.py`, `main_финал2.py`,
`main_точно_финал.py` — и через день тонешь в этом болоте. Нет истории, нет
возможности откатиться, страшно что-либо менять.

### Решение

**Git** — система контроля версий. Это «машина времени» для кода: он сохраняет
снимки проекта (коммиты), и к любому из них можно вернуться. История изменений —
на месте, эксперименты — без страха. (Само слово *git* — не сокращение, а просто
название, которое дал инструменту создатель Linux Линус Торвальдс.)

Сначала проверь, что git вообще установлен:
```powershell
git --version
```
Если команда не найдена — поставь git с [git-scm.com](https://git-scm.com/downloads).

Если ты пользуешься git впервые, представься ему один раз командой `git config`
(*configuration* — «настройка»); флаг `--global` означает «для всех проектов
сразу»:
```powershell
git config --global user.name "Твоё Имя"
git config --global user.email "ты@example.com"
```

Теперь внутри папки `python-trainer` заведи репозиторий командой `git init`
(*initialize* — «создать с нуля»):
```powershell
git init
```

Проверим состояние репозитория командой `git status` (*status* — «состояние»):
```powershell
git status
```
Git ответит что-то вроде *«On branch main / No commits yet»* или
*«On branch master / No commits yet»*. Это значит: репозиторий создан, но мы ещё
ничего не сохраняли. Позже мы всё равно назовём главную ветку `main`.

> **Важно:** первый коммит мы делать пока не будем. Сначала появятся файлы,
> которые в репозиторий попадать не должны (виртуальное окружение). Сначала
> создадим проект и `.gitignore`, и только потом — первый коммит. Иначе мы
> случайно «сфотографируем» тысячи лишних файлов.

`git init` создал скрытую папку `.git` — это и есть «хранилище машины времени».
Удалишь её — потеряешь всю историю. Трогать её руками не нужно.

## 1.4. Виртуальное окружение (venv)

> Это ключевой подраздел шага. Здесь — самая частая боль новичков, поэтому
> разберём её подробно.

Представь, что Django ты ставишь глобально, на весь компьютер. И вот ситуации,
в которые ты гарантированно вляпаешься:

1. **Конфликт версий между проектами.** Сегодня ты пишешь учебный проект на
   Django 6. Через месяц приходишь на работу/стажировку, а там старый проект на
   Django 4. Версия в системе одна. Чтобы запустить рабочий проект, тебе придётся
   удалить свой Django 6 и поставить Django 4. Закончил — ставь обратно 6. И так
   каждый раз. Это та самая «удалять-устанавливать разные версии», ад.
2. **«У меня работает, а у тебя нет».** На твоём компьютере случайно стоит нужная
   библиотека (ты её ставил для другого проекта). У товарища её нет. Один и тот же
   код у вас ведёт себя по-разному, и непонятно почему.
3. **Невозможно зафиксировать зависимости.** Если всё свалено в один глобальный
   Python, ты не знаешь, что именно нужно *этому* проекту, а что прилетело от
   других. А значит, не можешь честно сказать новому человеку: «поставь вот это».
4. **Замусоривание системного Python.** Сам Python в системе (а на macOS/Linux —
   ещё и системные утилиты!) полагается на определённые пакеты. Ставя и удаляя
   что попало глобально, легко сломать что-то постороннее.
5. **Права администратора.** Глобальная установка нередко требует прав админа и
   ругается ошибками доступа. В своей папке проекта таких проблем нет.
6. **Грязь сложно убрать.** Наигрался с проектом, удалил папку — а пакеты от него
   остались висеть в системе навсегда. С venv достаточно удалить одну папку.

### Решение

**Виртуальное окружение (venv)** — это отдельное окружение со своим Python и
своими библиотеками внутри папки проекта. У каждого проекта — своё окружение.
Поставил в него Django 6 — это никак не влияет на другие проекты. Проект с
Django 4 живёт в своём окружении и не конфликтует. Удалил его — стало чисто.

Создай виртуальное окружение в папке `.venv`. Здесь `venv` — *virtual
environment* («виртуальное окружение»), а флаг `-m` (*module* — «модуль») велит
Python запустить встроенный модуль с таким именем:
```powershell
python -m venv .venv
```

Активируй его (это самая «капризная» по системам команда). `activate` означает
«активировать», а расширение файла подсказывает его тип: `.ps1` — скрипт
PowerShell, `.bat` — пакетный файл команд Windows:

- **Windows / PowerShell:**
  ```powershell
  .venv\Scripts\Activate.ps1
  ```
- **Windows / cmd.exe:**
  ```cmd
  .venv\Scripts\activate.bat
  ```
- **macOS/Linux:** (`source` — встроенная команда «выполнить скрипт в текущей
  оболочке»)
  ```bash
  source .venv/bin/activate
  ```

> **Если PowerShell ругается** *«выполнение сценариев отключено в этой системе»*
> (`running scripts is disabled`) — это защита Windows. На время текущего окна
> терминала можно разрешить запуск локальных скриптов командой `Set-ExecutionPolicy`
> («задать политику запуска скриптов»):
> ```powershell
> Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
> ```
> `-Scope Process` означает «только для этого запущенного PowerShell». Закроешь
> окно — настройка исчезнет. После этого снова выполни команду активации.

После активации в начале строки терминала появится приставка `(.venv)`:
```
(.venv) PS C:\Users\ты\python-trainer>
```
Это значит, что мы внутри окружения. Проверим, что Python теперь берётся именно
из неё:
```powershell
# Windows PowerShell — Get-Command «показывает, откуда берётся команда»
Get-Command python
```
```bash
# macOS/Linux — which «который», то есть «где лежит эта команда»
which python
```
Путь должен указывать внутрь `.venv` твоего проекта.

Посмотрим командой `python -m pip list` («покажи список установленного»), что
пока в окружении почти пусто:
```powershell
python -m pip list
```

> **Запомни главное правило:** venv нужно активировать заново каждый раз, когда ты
> открыл новый терминал и сел работать над проектом. Признак, что всё ок, —
> приставка `(.venv)` в строке. Чтобы выйти из окружения, есть команда
> `deactivate` («деактивировать»).

`venv` — это просто папка `.venv` с копией интерпретатора Python и отдельной
папкой для библиотек. «Активация» лишь временно говорит терминалу: «когда пишут
`python` и `pip`, бери их вот отсюда, а не из системы». Никакой магии — поэтому
эту папку мы потом запретим коммитить (она огромная и легко пересоздаётся).

## 1.5. Устанавливаем Django

Чистый Python не умеет «в веб» из коробки. Чтобы принять запрос из браузера,
разобрать адрес, найти нужный обработчик и вернуть HTML-страницу, на голом Python
пришлось бы писать сотни строк низкоуровневого кода. HTML (*HyperText Markup
Language*) — язык разметки страницы: заголовки, абзацы, ссылки, формы. Без
помощников пришлось бы самим описывать, как браузер и сервер передают друг другу
сообщения, как понять адрес страницы, как собрать ответ. Это долго, скучно и
полно ошибок.

### Решение

**Django** — веб-фреймворк. Он берёт на себя всю рутину: приём запросов,
маршруты, шаблоны, работу с базой данных и админку — готовую служебную часть для
управления данными. Нам остаётся интересное — логика нашего тренажёра. Ставим
Django в активированное окружение через `pip` (*Package Installer for Python* —
«установщик пакетов для Python»).

Убедись, что в строке есть `(.venv)`, и поставь Django командой
`python -m pip install` (`install` — «установить»):
```powershell
python -m pip install "Django>=6.0,<6.1"
```
Так мы запускаем `pip` через тот же Python, который находится внутри `.venv`.
Ограничение `"Django>=6.0,<6.1"` означает: «поставь Django версии 6.0, но не
переходи на будущую 6.1». В учебном курсе это важно: команды и выводы остаются
предсказуемыми.

Проверим версию (`python -m django` — «запусти модуль django»):
```powershell
python -m django --version
```
Должна напечататься версия, например `6.0.6`. Поздравляю — Django в проекте.

> **Почему именно в venv?** Перечитай: Django поставился в окружение `.venv`, а не в
> систему. Другой проект с другой версией Django об этом даже не узнает. Та самая
> боль из 1.4 — вылечена.

`python -m pip` скачал Django из общего хранилища пакетов PyPI (*Python Package
Index* — «каталог пакетов Python», pypi.org) и положил его внутрь `.venv`. Вместе
с Django подтянулись пакеты, от которых он зависит (например, `asgiref` и
`sqlparse`) — это нормально.

## 1.6. requirements.txt — список зависимостей

Саму папку `.venv` мы в git не кладём (она огромная и у каждого своя). Но тогда
вопрос: товарищ (или ты сам на другом компьютере) клонирует репозиторий — и как
ему узнать, *что* нужно поставить, чтобы проект заработал? «Помню, ставил
Django… и что-то ещё, кажется?» — так не пойдёт.

### Решение

Файл **`requirements.txt`** (*requirements* — «требования, зависимости») — это
список зависимостей проекта с версиями. По нему любой одной командой воссоздаёт
точно такое же окружение.

Создаём его командой `python -m pip freeze` (*freeze* — «заморозить,
зафиксировать список»); символ `>` перенаправляет вывод команды в файл:
```powershell
python -m pip freeze > requirements.txt
```

Открой `requirements.txt` — внутри будет что-то вроде:
```
asgiref==3.9.1
Django==6.0.6
sqlparse==0.5.3
```
Так выглядит «снимок» окружения: здесь есть сам Django и пакеты, которые Django
поставил вместе с собой. Позже любой сможет повторить его командой
`python -m pip install -r requirements.txt` (флаг `-r` от *requirements* —
«прочитать список из файла»), запущенной внутри своего свежего venv.

## 1.7. Создаём Django-проект

Чтобы Django заработал, нужна определённая структура файлов: точка входа,
настройки, таблица маршрутов, файлы для запуска на сервере. Создавать всё это
руками — долго и легко ошибиться в одной запятой.

### Решение

У Django есть генератор-«заготовщик»: команда `startproject` (*start project* —
«создать проект») создаёт правильную структуру за нас. `django-admin` — служебная
утилита Django для административных команд (*admin* — администрирование).

Обрати внимание на точку в конце команды:
```powershell
django-admin startproject trainer_project .
```

> **Зачем точка `.`?** Она означает «создавай проект в текущей папке». Без точки
> Django создаст лишнюю обёртку: `python-trainer/trainer_project/...` — папка в
> папке с одинаковым смыслом, в которой все потом путаются. Точка избавляет от
> этого матрёшечного вложения.

Посмотри, что появилось:
```powershell
ls
```
Структура теперь такая:
```
python-trainer/
├── .venv/                  ← наше окружение (в git не пойдёт)
├── manage.py               ← пульт управления проектом
├── requirements.txt
└── trainer_project/        ← пакет с настройками проекта
    ├── __init__.py
    ├── asgi.py
    ├── settings.py         ← все настройки проекта
    ├── urls.py             ← таблица маршрутов (какой адрес → какой код)
    └── wsgi.py
```

Сразу попросим Django проверить настройки командой `check` («проверить»):
```powershell
python manage.py check
```
Если всё в порядке, Django напишет `System check identified no issues`.

Запусти встроенный сервер разработки командой `runserver` (*run server* —
«запустить сервер»):
```powershell
python manage.py runserver
```
Открой в браузере адрес из вывода — http://127.0.0.1:8000/. Ты увидишь
страницу-«ракету» Django: *«The install worked successfully! Congratulations!»*.
Это значит, что веб-сервер поднялся и работает.

Останови сервер сочетанием `Ctrl+C` в терминале.

> **Заметил красную надпись?** При запуске Django пишет:
> *«You have N unapplied migration(s)…»* (есть непринятые миграции). Не пугайся и
> пока ничего не делай — это след будущей боли, к которой мы специально подойдём в
> конце шага.

Что за файлы появились:
- `manage.py` (*manage* — «управлять») — «пульт» проекта. Через него мы запускаем
  команды: `runserver`, создание приложений, работу с базой данных и т. д.
- `settings.py` (*settings* — «настройки») — какие приложения подключены, какая
  база данных, язык, часовой пояс, секретный ключ.
- `urls.py` (*URL* — адрес страницы) — таблица маршрутов: какому адресу какой код
  отвечает.
- `wsgi.py` / `asgi.py` — точки входа для «боевого» сервера. WSGI — *Web Server
  Gateway Interface*, ASGI — *Asynchronous Server Gateway Interface* (асинхронный
  вариант). Понадобятся при публикации в интернет; сейчас просто лежат.

> **Про секретный ключ.** В `settings.py` Django создал `SECRET_KEY` — секретный
> ключ проекта. Сейчас это учебный проект для локальной разработки, поэтому мы
> временно оставляем файл как есть. В настоящем опубликованном сервисе такой ключ
> нельзя хранить в git: его выносят в `.env` или в настройки сервера. К этой боли
> мы вернёмся перед первым настоящим размещением проекта в интернете.

## 1.8. .gitignore — что НЕ коммитить

Если сейчас сделать `git add .`, в репозиторий попадёт всё подряд, в том числе:
- папка `.venv` — это тысячи файлов; репозиторий мгновенно распухнет, а файлы эти
  у каждого свои и легко пересоздаются (по `requirements.txt`);
- файл базы данных `db.sqlite3` — это локальные данные, а не код. SQLite —
  встроенная в Python маленькая база данных в одном файле;
- папки `__pycache__/` — служебный кэш Python, мусор;
- в реальных проектах — секреты и пароли, которые категорически нельзя
  публиковать.

Git должен следить только за исходным кодом, а не за мусором и секретами.

### Решение

Файл **`.gitignore`** (*ignore* — «игнорировать») — список того, что git должен
не замечать. У GitHub есть готовый, проверенный шаблон для Python-проектов —
возьмём его.

Создай в корне проекта файл `.gitignore`. Содержимое можно взять из официального
шаблона GitHub: [github/gitignore → Python.gitignore](https://github.com/github/gitignore/blob/main/Python.gitignore).
Главное, чтобы внутри точно были эти строки:
```gitignore
# Виртуальное окружение
.venv/
venv/

# Кэш Python
__pycache__/
*.py[cod]

# База данных SQLite (локальные данные)
db.sqlite3
db.sqlite3-journal

# Секреты
.env
```

Проверим, что игнор работает:
```powershell
git status
```
Теперь в списке не должно быть `.venv/` и `db.sqlite3`. Git «видит» `manage.py`,
папку `trainer_project/`, `.gitignore`, `requirements.txt` — то, что нам и нужно
сохранить.

`.gitignore` не удаляет файлы — он лишь говорит git: «делай вид, что этих файлов
нет, не предлагай их сохранять». Файлы остаются на диске и работают как обычно.

## 1.9. Первый коммит и публикация на GitHub

Сейчас весь проект — только на твоём компьютере. Полетит диск, зальёшь кофе на
ноутбук, потеряешь его — и весь труд исчезнет. Плюс: код негде показать
наставнику, не из чего собрать портфолио, нельзя работать с другого устройства.

### Решение

**GitHub** — облачный «дом» для git-репозиториев. Это резервная копия, витрина
проекта и точка совместной работы. Сделаем первый коммит (снимок состояния) и
запушим его на GitHub.

**1) Сделай первый коммит** (локально). Сначала посмотри, что git видит:
```powershell
git status
```

Теперь добавь файлы в будущий снимок и проверь, что именно в него войдёт:
```powershell
git add .
git diff --staged --stat
```

Если в списке нет `.venv/`, `db.sqlite3` и других лишних файлов, делай коммит:
```powershell
git commit -m "Initial commit: пустой Django-проект"
```

**2) Создай пустой репозиторий на GitHub:**
зайди на [github.com](https://github.com), нажми `New repository`, назови его
`python-trainer`. Не добавляй при создании README/.gitignore (они у нас уже
есть) — иначе будет конфликт. Нажми `Create`.

**3) Привяжи свой локальный репозиторий к GitHub и отправь код.** GitHub после
создания покажет точные команды; они выглядят так (подставь свой логин):
```powershell
git branch -M main
git remote add origin https://github.com/ТВОЙ_ЛОГИН/python-trainer.git
git push -u origin main
```

> **Про вход.** При первом `push` GitHub попросит авторизоваться. Самый простой
> путь на Windows — установить [GitHub CLI](https://cli.github.com/) (CLI —
> *Command Line Interface*, «интерфейс командной строки») и выполнить
> `gh auth login`, либо войти через всплывающее окно Git Credential Manager.
> Обычный пароль от аккаунта в терминале не подойдёт — нужен токен или вход через
> браузер, это нормально.

Обнови страницу репозитория на GitHub — там появятся твои файлы (`manage.py`,
папка `trainer_project/`, `.gitignore`, `requirements.txt`). Папки `.venv` и файла
`db.sqlite3` там быть не должно — это значит, `.gitignore` отработал верно.

Что означают команды, которые мы выполнили:
- `git add .` (*add* — «добавить», точка `.` — «все файлы в текущей папке») —
  помечает, что войдёт в снимок.
- `git diff --staged --stat` (*difference* — «разница», *staged* — «подготовлено»,
  *statistics* — «статистика») — показывает короткий список файлов, уже
  подготовленных к коммиту.
- `git commit -m` (*commit* — «зафиксировать», `-m` от *message* — «сообщение») —
  делает сам снимок проекта с короткой подписью.
- `git branch -M main` (*branch* — «ветка», `-M` от *move* — здесь
  «переименовать») — называет главную ветку `main`.
- `git remote add origin` (*remote* — «удалённый») — записывает адрес репозитория
  на GitHub под именем `origin` («источник» — общепринятое имя).
- `git push -u` (*push* — «отправить», `-u` = `--set-upstream`) — отправляет
  коммиты на GitHub и запоминает связь, чтобы дальше хватало просто `git push`.

## 1.10. Создаём приложение (app)

У нас есть `trainer_project` — но это «центр управления» (настройки, общие
маршруты). А где писать саму логику сайта: страницы, задачи тренажёра? Если
свалить всё в папку настроек, она быстро превратится в помойку. По мере роста
проекта в нём появятся разные части (задачи, профиль, рейтинг) — их надо как-то
разделять.

### Решение

В Django проект состоит из **приложений (apps)** — отдельных модулей под свою
часть функциональности. Так код остаётся опрятным, а части — переиспользуемыми.
Создадим приложение `trainer_app` под основную логику тренажёра.

Создаём приложение командой `startapp` (*start app*, где *app* — *application* —
«создать приложение»):
```powershell
python manage.py startapp trainer_app
```

Теперь зарегистрируй приложение, иначе проект про него «не знает». Открой
`trainer_project/settings.py`, найди список `INSTALLED_APPS` («установленные
приложения») и добавь в конец путь к настройке приложения:
```python
INSTALLED_APPS = [
    'django.contrib.admin',
    'django.contrib.auth',
    'django.contrib.contenttypes',
    'django.contrib.sessions',
    'django.contrib.messages',
    'django.contrib.staticfiles',
    'trainer_app.apps.TrainerAppConfig',  # ← наше приложение
]
```
`TrainerAppConfig` — готовый класс настройки приложения, который Django создал в
`trainer_app/apps.py`. Так мы подключаем приложение явно: не просто «папку
`trainer_app`», а её официальную настройку.

Появилась папка `trainer_app/`:
```
trainer_app/
├── __init__.py
├── admin.py        ← настройка админки
├── apps.py         ← конфигурация приложения
├── migrations/     ← история изменений базы данных (пока пусто)
├── models.py       ← модели данных (пока пусто!)
├── tests.py        ← тесты
└── views.py        ← обработчики страниц (views)
```

> **Зачем регистрировать в `INSTALLED_APPS`?** Это та самая боль: пока приложение
> не в списке, Django его не подхватит — не найдёт ни его шаблоны, ни (позже)
> модели. Регистрация — способ сказать проекту: «это приложение — часть меня».

Загляни в `models.py` — там пусто, только строчка `# Create your models here.`.
Запомни это место. Именно сюда мы вернёмся в Шаге 2, когда научим тренажёр
хранить данные. Пока оставляем как есть.

## 1.11. Первая своя страница: view + url + шаблон

Это самый «составной» подраздел: чтобы на экране появилась наша страница, нужны
три кусочка, которые работают в связке. Введём их по очереди — каждый в ответ на
свою боль.

Как Django обрабатывает запрос (запомни эту цепочку):

```
браузер → urls.py (какой адрес?) → views.py (что ответить?) → шаблон (как выглядит) → браузер
```

### 1.11.1. View — функция-обработчик

Сервер работает, но на главной — стандартная «ракета» Django. Нам нужна своя
страница. Кто-то должен на запрос браузера сформировать ответ. Этого «кого-то» в
Django называют **view** (англ. «представление») — функция, которая получает
запрос и возвращает ответ.

#### Решение

Напишем функцию-view `home` («главная»), которая будет отвечать на запрос главной
страницы. Открой `trainer_app/views.py` и приведи к виду:
```python
from django.shortcuts import render


def home(request):
    return render(request, 'trainer_app/index.html')
```
`home` принимает запрос (`request` — «запрос») и через `render` (англ.
«отрисовать, собрать страницу») возвращает шаблон `index.html`. Самого шаблона
ещё нет — создадим его следующим.

### 1.11.2. Шаблон — как страница выглядит

Можно было бы вернуть HTML прямо строкой из Python, но это быстро превращается в
кашу: разметка, экранирование кавычек, никакой подсветки. Смешивать логику и
вёрстку — больно.

#### Решение

**Шаблоны (templates)** — отдельные HTML-файлы. Django сам их находит и
отрисовывает. Логика — в Python, внешний вид — в HTML.

Внутри приложения создай папку и файл:
`trainer_app/templates/trainer_app/index.html`

```html
<!DOCTYPE html>
<html lang="ru">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Python-тренажёр</title>
</head>
<body>
    <h1>Сайт находится в разработке</h1>
    <p>Зайдите к нам позже.</p>
</body>
</html>
```

> **Зачем такая «двойная» вложенность `templates/trainer_app/`?** Это не опечатка.
> Django ищет шаблоны во всех приложениях и складывает их в общую кучу. Если в
> двух приложениях окажется по `index.html`, он не поймёт, какой брать. Папка с
> именем приложения внутри `templates/` делает имя уникальным:
> `trainer_app/index.html`. Это лечит будущую боль конфликта имён.

### 1.11.3. URL приложения — связать адрес и view

View написан, шаблон есть — но Django пока не знает, по какому адресу вызывать
`home`. Нужна связь «адрес → функция».

#### Решение

Своя таблица маршрутов внутри приложения — файл `urls.py`. Создай файл
`trainer_app/urls.py`:
```python
from django.urls import path
from . import views

urlpatterns = [
    path('', views.home, name='home'),
]
```
Здесь `path` (англ. «путь, маршрут») связывает адрес с функцией, а список
`urlpatterns` («шаблоны адресов») — это все маршруты приложения. Пустой адрес
`''` — это «корень», то есть главная страница.

### 1.11.4. Подключить маршруты приложения к проекту

У приложения теперь есть свои маршруты, но проект про них не знает — главный
`urls.py` ничего не подключает. Запрос просто не дойдёт до нашего `home`.

#### Решение

В главном `urls.py` проекта подключим маршруты приложения через `include()`
(англ. «включить, подключить»). Открой `trainer_project/urls.py` и приведи к виду:
```python
from django.contrib import admin
from django.urls import path, include

urlpatterns = [
    path('admin/', admin.site.urls),
    path('', include('trainer_app.urls')),
]
```
Строка с `include` означает: «все адреса, начинающиеся с корня, передавай в
`trainer_app/urls.py`».

Запусти сервер:
```powershell
python manage.py runserver
```
Открой http://127.0.0.1:8000/ — теперь там наша страница «Сайт находится в
разработке». Это первый живой результат: любой посетитель видит, что сайт работает.
Останови сервер `Ctrl+C`.

Попроси Django ещё раз проверить проект:
```powershell
python manage.py check
```
Ответ `System check identified no issues` означает, что Django не видит ошибок в
настройках, приложениях и маршрутах.

Полная цепочка для главной страницы сложилась:
```
браузер запросил "/"
   → trainer_project/urls.py: "/" отдаём в trainer_app.urls (include)
   → trainer_app/urls.py: "" → views.home
   → views.home: render шаблона trainer_app/index.html
   → браузер показал «Сайт находится в разработке»
```

Мы проделали кучу работы (приложение, view, шаблон, маршруты), но всё это пока
только на диске. Машина времени про эти изменения ещё не знает — а значит, они не
защищены и их нет на GitHub.

### Решение

Сначала посмотрим, какие файлы изменились:
```powershell
git status
```

Подготовим изменения к коммиту и проверим короткий список файлов:
```powershell
git add .
git diff --staged --stat
```

Если там только ожидаемые файлы приложения, сохраняем прогресс новым коммитом и

```powershell

```

Обнови репозиторий на GitHub — там появилось приложение `trainer_app/` с view,
шаблоном и маршрутами. Шаг 1 завершён.

## Новая боль → Шаг 2: база данных

Оглянись — у нас работающий сайт. Но он ничего не помнит и почти ничего не умеет
хранить. Конкретные боли, с которыми мы упёрлись:

1. **Тренажёр пока статичен.** Все «задачи» пришлось бы вписывать прямо в HTML.
   Но мы хотим: добавлять новые задачи без правки кода, помнить, какие задачи
   решил пользователь, считать его результат. Обычные переменные Python для этого
   не годятся — они исчезают при перезапуске сервера. Данные должны где-то жить
   *постоянно*.

2. **Красная надпись про миграции.** Помнишь *«You have N unapplied migrations»*
   при каждом запуске? Это Django намекает: у него уже есть таблицы (для
   пользователей, админки, сессий), которые надо «развернуть» в базе данных. Мы их
   так и не тронули.

3. **Пустой `models.py`.** Помнишь строчку `# Create your models here.`? Вот мы и
   подошли к моменту, когда пора создавать модели.

Решение всех трёх болей — одно: база данных и модели Django. Этим и займёмся в
Шаге 2: опишем первую модель (например, «Задача»), познакомимся с **миграциями**,
применим их и впервые сохраним данные, которые переживут перезапуск.
