## 2.0. Что строим в этом шаге

Наш сервис — тренажёр для совместного обучения. Прежде чем городить задачи,
профили и подбор напарников, сделаем маленькую, но настоящую фичу для главной

Идти будем как всегда — от боли. Сначала решим задачу самым наивным способом

## 2.1. Первая цитата на главной из файла

Главная сейчас — заглушка «Сайт находится в разработке». А у нас уже есть готовые
цитаты в `quotes.txt`. Самый прямой способ показать цитату — прочитать файл прямо
в обработчике страницы и отдать случайную строку в шаблон.

### Решение

Положи файл `quotes.txt` в корень проекта, рядом с `manage.py`. Внутри — по одной
цитате на строку, текст и автор разделены чертой `|`:
```
Корни образования горькие, но плоды сладкие.|Аристотель
Нет ничего постоянного, кроме изменений.|Гераклит
Даже путь в тысячу ли начинается с первого шага.|Лао-Цзы
Всё течёт, всё меняется. И никто не был дважды в одной реке. Ибо через миг и река была не та, и сам он уже не тот.|Гераклит
Два года человек учится говорить, а потом всю оставшуюся жизнь — молчать.|Аристотель
Самый лучший способ жить с честью в этом мире - быть тем, кем мы притворяемся.|Сократ
Щедрый человек — это тот, кто дает подходящему человеку подходящую вещь в подходящее время.|Аристотель
Я знаю, что ничего не знаю, но другие не знают и этого.|Сократ
Скажи мне — и я забуду, покажи мне — и я запомню, дай мне сделать — и я пойму.|Конфуций
Одни копят, словно должны жить вечно, другие тратят, словно тотчас умрут.|Аристотель
Голод — лучшая приправа к пище.|Сократ
Природа обыскивает нас при выходе, как при входе. Нельзя вынести больше, чем принес.|Сенека
Лучший способ начать дружбу - это помочь, лучший способ продолжить - не просить взамен ничего.|Сократ
Кто знает - не говорит. Кто говорит - не знает.|Лао-Цзы
Побороть дурные привычки можно только сегодня, а не завтра.|Конфуций
Самая большая помеха жизни — ожидание завтрашнего дня и потеря сегодняшнего дня.|Сенека
Если бы у слепого спросили, что такое зрение, то он ответил бы, что это слепота.|Гераклит
Красота есть во всем, но не всем дано это видеть.|Конфуций
Если человек не знает куда он плывет, то для него нет попутного ветра.|Сенека
Если человек долго поднимался по чужой лестнице, то, прежде чем найти свою, ему нужно спуститься.|Лао-Цзы
```

Теперь научим view читать этот файл. Открой `trainer_app/views.py` и приведи к
виду:
```python
import random

from django.conf import settings
from django.shortcuts import render


def home(request):
    quotes_file = settings.BASE_DIR / 'quotes.txt'
    lines = quotes_file.read_text(encoding='utf-8').splitlines()
    text, author = random.choice(lines).split('|')
    return render(request, 'trainer_app/index.html', {'text': text, 'author': author})
```
Разберём по строчкам. `settings.BASE_DIR` — это не просто текст с путём, а объект
`Path` из стандартного модуля `pathlib` (*path* — «путь»); Django хранит в нём
корень проекта. У путей-объектов `Path` черта `/` перегружена: она не делит числа, а
*склеивает* части пути. Поэтому `settings.BASE_DIR / 'quotes.txt'` — это путь до
нашего файла, а значит, `quotes_file` — тоже объект `Path`, указывающий на
`quotes.txt`.

У объекта `Path` есть удобный метод `read_text` (*read text* — «прочитать текст»):
он сам открывает файл, читает его целиком в одну строку и закрывает — не нужно
вызывать ни `open`, ни `close`. Аргумент `encoding='utf-8'` говорит, в какой
кодировке читать, чтобы русские буквы не превратились в кракозябры. Дальше строковый
метод `splitlines` (*split lines* — «разбить на строки») режет этот большой текст на
список строк, по строке на цитату; `random.choice` берёт из списка случайную, а
`split('|')` делит её на текст и автора. Последним аргументом `render` мы передаём в
шаблон словарь с данными — это называется контекстом.

Покажем цитату в шаблоне. Открой `trainer_app/templates/trainer_app/index.html` и
замени содержимое `<body>`:
```html
<body>
    <blockquote>{{ text }}</blockquote>
    <p>— {{ author }}</p>
</body>
```
Двойные фигурные скобки `{{ text }}` — это место, куда Django подставит значение из
контекста.

Запусти сервер и обнови страницу несколько раз:
```powershell
python manage.py runserver
```
Открой http://127.0.0.1:8000/ — на каждой перезагрузке появляется случайная цитата
с автором. Победа: главная ожила. Останови сервер `Ctrl+C`.

## 2.2. Почему файл плохой дом для данных

Работает же — зачем что-то ещё? Давай честно посмотрим, во что мы упрёмся, как
только цитат станет много и ими захочется управлять.

Чтобы добавить или убрать цитату, нужно лезть в файл руками и перезапускать сайт —
а значит, это умеет только программист, и только с доступом к коду.
Контент-редактор (человек, которого наняли для наполнения сайта содержимым) так не
сможет. Разделитель `|` хрупкий: появится цитата, где сам символ `|` внутри текста,
— и `split('|')` развалится. У строк нет ни номера, ни порядка, ни служебных данных
вроде «кто добавил и когда». Нельзя быстро найти все цитаты одного автора,
отсортировать, посчитать. И на каждый заход на страницу мы перечитываем весь файл
целиком.

### Решение

Всё это — работа для **базы данных**: отдельного хранилища, которое держит данные
в виде таблиц со строками и колонками, умеет искать, сортировать, добавлять и
удалять записи и переживает перезапуск сервера. Данные перестают быть частью кода —
и ими можно управлять, не трогая код.

Django уже всё подготовил для базы: вспомни красную надпись из Шага 1 —
*«You have N unapplied migrations»* — и файл `db.sqlite3`, который мы внесли в
`.gitignore`. Это и есть наша база (SQLite — маленькая база в одном файле, ты с ней
уже знаком). Слово **миграции** здесь означает инструкции для базы: какие таблицы
нужно создать или изменить, чтобы код Django и база говорили на одном языке. Эту
базу Django завёл заранее, но та самая красная надпись так и висит при каждом
запуске: что-то ещё не доделано. Закрыть её — первое дело следующего подшага.

## 2.3. Накатываем миграции Django

База-файл `db.sqlite3` пока пустой: в нём нет ни одной таблицы. Django при каждом
запуске напоминает об этом красной надписью про «непринятые миграции». У самого
Django есть встроенные приложения (пользователи, сессии, админка), которым нужны
свои таблицы, — и они ещё не созданы.

### Решение

**Миграция** — это файл с пошаговой инструкцией для базы: какие таблицы создать,
какие колонки в них добавить, что переименовать или удалить. Важно: миграция хранит
не сами данные (не цитаты), а план устройства хранилища — как чертёж полок, на
которые данные потом будут складываться.

Зачем нужен отдельный файл, а не команды к базе напрямую? Чтобы код Django и база
всегда сходились. По твоему коду Django знает, какие таблицы ему нужны; если в базе
их ещё нет, он и выводит ту самую красную надпись. Миграция — мостик между ними: она
переносит изменение из кода в саму базу (само слово «миграция» — про перенос), и
надпись гаснет.

Как миграция выглядит: это маленький Python-файл с номером в имени, вроде
`0001_initial.py`, лежащий в папке `migrations/` внутри приложения. У встроенных
приложений Django (пользователи, сессии, админка) такие файлы заготовлены заранее;
свою первую миграцию мы создадим и откроем, чтобы заглянуть внутрь, в подшаге 2.5.

Пока нам осталось применить готовые миграции встроенных приложений — командой
`migrate` (*migrate* — «перенести, применить»):
```powershell
python manage.py migrate
```
Django пробежится по заготовленным миграциям и создаст нужные таблицы. В конце
увидишь столбик строк `OK`.

Запусти сервер — красной надписи про миграции больше нет:
```powershell
python manage.py runserver
```
Останови его `Ctrl+C`.

Заглянуть в результат можно знакомым тебе DBeaver: открой файл `db.sqlite3` как
базу SQLite — внутри появились таблицы вроде `auth_user`, `django_session` и
другие. Их структуру Django создал за тебя; ни одной строки SQL ты не написал. Это
первая встреча с тем, ради чего весь шаг.

## 2.4. Модель и ORM вместо ручного SQL

Встроенные таблицы есть, но таблицы под наши цитаты — нет. Раньше, на голом
`sqlite3`, ты завёл бы её руками: написал бы `CREATE TABLE quote (id INTEGER
PRIMARY KEY, text TEXT, author TEXT)`, потом вставлял бы строки запросами `INSERT`,
следил за типами, экранированием и SQL-инъекциями. Знакомая рутина — и на ней легко
ошибиться.

### Решение

Django предлагает другой путь — **ORM** (*Object-Relational Mapping* — «отображение
объектов на реляционную базу»). Идея: ты описываешь таблицу как обычный
Python-класс — **модель**, а Django сам переводит её в SQL и обратно. Таблица — это
класс, строка таблицы — объект этого класса, колонка — атрибут. SQL руками писать
больше не нужно.

Помнишь пустой `models.py` с подписью `# Create your models here.` из Шага 1? Вот
мы и пришли. Открой `trainer_app/models.py` и опиши модель цитаты:
```python
from django.db import models


class Quote(models.Model):
    text = models.TextField()
    author = models.CharField(max_length=100)
```
Разберём по словам. `models.Model` — базовый класс всех моделей Django; наследуя
его, `Quote` получает умение сохраняться в базу. `TextField` — длинный текст без
ограничения длины (для самой цитаты). `CharField` — короткая строка, и для неё
Django требует `max_length` — предельную длину (для имени автора хватит 100).
Колонку `id` — автонумеруемый первичный ключ — Django добавит сам, её описывать не
надо.

Ты уже умеешь работать с классами и атрибутами — здесь ровно то же самое, только
атрибуты-поля станут колонками таблицы.

## 2.5. makemigrations и migrate для Quote

Класс `Quote` мы написали, но в базе таблицы под него ещё нет: Django пока не знает,
что модель изменилась. Между «поправил код модели» и «в базе появилась таблица»
есть шаг — и это снова миграции.

### Решение

Сначала Django должен заметить изменение и записать план. Команда `makemigrations`
(*make migrations* — «создать миграции») смотрит на твои модели и пишет файл-инструкцию:
```powershell
python manage.py makemigrations trainer_app
```
Django создаст файл `trainer_app/migrations/0001_initial.py`. Открой его — внутри
на Python описано `migrations.CreateModel(...)`: какая таблица и какие поля. Это
ещё не сама база, а план будущего изменения.

Теперь применим план к базе уже знакомой командой:
```powershell
python manage.py migrate
```
Django выполнит план и создаст таблицу `trainer_app_quote`. И снова: ни строки SQL
руками — `CREATE TABLE` Django написал и выполнил за тебя по твоему Python-классу.
Открой `db.sqlite3` в DBeaver: таблица `trainer_app_quote` с колонками `id`, `text`,
`author` на месте.

Запомни рабочий цикл: поменял модель → `makemigrations` (записать план) → `migrate`
(применить). Файлы в папке `migrations/` — это версии структуры базы, по сути
«машина времени» для схемы, как git для кода. Их тоже коммитят в репозиторий.

## 2.6. Управляем данными через админку

Таблица `trainer_app_quote` есть, но пустая, и положить в неё цитату пока нечем —
разве что лезть в код. А управлять контентом хочется просто: добавил, поправил,
удалил — и желательно мышкой, без программирования. Тогда данными сможет заниматься
не только программист.

### Решение

У Django панель управления данными — **админка** — уже встроена, осталось включить
её для нашей модели. Открой `trainer_app/admin.py`:
```python
from django.contrib import admin

from .models import Quote

admin.site.register(Quote)
```
`admin.site.register` означает «показывать эту модель в админке».

Чтобы войти, нужен администратор. Создай его командой `createsuperuser` (*create
superuser* — «создать суперпользователя»):
```powershell
python manage.py createsuperuser
```
Django спросит имя, почту (можно пропустить) и пароль. Запусти сервер, открой
http://127.0.0.1:8000/admin/ и войди. В разделе «Quotes» нажми «Add» и добавь пару
цитат мышкой — без единой строки кода. Готовую CRUD-панель
(создать-прочитать-обновить-удалить) ты получил даром, просто описав модель: вот
что Django умеет из коробки.

Останови сервер `Ctrl+C`. Одно только режет глаз: в списке цитаты подписаны как
`Quote object (1)`, `Quote object (2)` — не разобрать, где какая. Это и починим
следующим шагом.

## 2.7. Делаем объект читаемым

В списке админки каждая цитата подписана как `Quote object (1)` — видно, что объект
есть, но не разобрать, какая это цитата. Та же беда поджидает и в терминале, куда мы
зайдём дальше.

### Решение

За то, как объект показывает себя текстом, отвечает метод `__str__`. Это служебный метод (функция), который мы видели в выводе функции `dir()`, когда изучали методы встроенных в Python классов. Добавим его в модель (наш класс `Quote`). Открой `trainer_app/models.py`:
```python
from django.db import models


class Quote(models.Model):
    text = models.TextField()
    author = models.CharField(max_length=100)

    def __str__(self):
        return f'{self.text[:40]}… — {self.author}'
```
Обнови список в админке — теперь каждая цитата подписана понятно: начало текста и
автор. `self.text[:40]` — первые 40 символов, чтобы длинная цитата не растягивала
строку.

Важная деталь: `__str__` меняет только поведение объекта в Python, а не структуру
таблицы. Поэтому миграция здесь не нужна — `makemigrations` тебе так и скажет: «No
changes detected». Миграции нужны лишь когда меняются поля.

## 2.8. ORM в терминале: Django shell

Через админку добавить одну цитату — приятно. Но любимых у нас два десятка в
`quotes.txt`, и кликать их по одной — мучение. Да и главная страница скоро должна
будет выбирать цитату *в коде*, а не мышкой. Значит, пора научиться говорить с
данными напрямую, из программы.

### Решение

С ORM данные — это объекты Python, а не SQL-строки. Раньше, на голом `sqlite3`,
тебя ждала связка «соединение → курсор → `execute` с SQL-строкой → `commit` →
закрыть». Теперь — просто вызовы методов. Django даёт оболочку, которая знает про
твой проект, — `shell`:
```powershell
python manage.py shell
```
Внутри (приглашение `>>>`) поработаем с цитатами через модель:
```python
from trainer_app.models import Quote

# добавить цитату — одна строка вместо INSERT
Quote.objects.create(text='Корни образования горькие, но плоды сладкие.', author='Аристотель')

# достать все цитаты
Quote.objects.all()

# посчитать
Quote.objects.count()

# отобрать по автору
Quote.objects.filter(author='Аристотель')

# взять одну
Quote.objects.first()
```
`Quote.objects` — менеджер модели, точка входа ко всем запросам. `create` делает
вставку одним вызовом, `all` возвращает все записи, `filter` отбирает по условию,
`count` считает, `first` берёт первую. Ни соединения, ни курсора, ни `commit`, ни
единой SQL-строки — Django превращает твои вызовы в SQL под капотом. Вспомни,
сколько кода это занимало на чистом `sqlite3`, — вот она, ценность ORM. И заметь:
цитаты, добавленные в админке, тут же видны через `Quote.objects.all()` — это одни
и те же данные, просто с разных сторон.

Выйти из оболочки — командой `exit()`.

## 2.9. Заливаем quotes.txt в базу

Класть цитаты по одной мы умеем — и в админке, и в shell. Но все 20 любимых уже
лежат в `quotes.txt`, и добавлять их по одной — мучение. Хочется одним махом перелить
весь файл в базу.

### Решение

Открой shell и пройдись по файлу циклом: каждая строка станет одной цитатой. Почти
всё здесь уже знакомо — путь к файлу, `read_text` и `splitlines` мы разбирали в 2.1,
а `Quote.objects` — в 2.8.
```powershell
python manage.py shell
```
```python
from django.conf import settings
from trainer_app.models import Quote

lines = (settings.BASE_DIR / 'quotes.txt').read_text(encoding='utf-8').splitlines()
for line in lines:
    text, author = line.split('|')
    Quote.objects.get_or_create(text=text, author=author)
```
Цикл `for` идёт по строкам файла, `split('|')` делит каждую на текст и автора. Новое
здесь одно — `get_or_create`: он либо находит такую цитату, либо создаёт новую.
Благодаря этому дубли не появятся, цикл можно запускать повторно, и цитаты, которые
ты добавил раньше руками, не задвоятся. Выйди из shell командой `exit()` и загляни в
админку — все 20 цитат на месте.

### 2.9.1. Необязательно: своя команда loadquotes

> **Необязательный подшаг — можно вернуться позже.** Цитаты уже в базе, и победа
> засчитана. Дальше кода станет заметно больше, и встретятся непривычные конструкции.
> Если они пока сбивают с толку, спокойно переходи к 2.10: на остальной шаг это не
> повлияет. А когда будешь готов — возвращайся, инструмент того стоит.

Цикл из 2.9 одноразовый: его придётся набирать заново в каждом новом shell и на
каждом компьютере, где появится проект. Хочется, чтобы заливка цитат стала постоянной
командой — такой же, как `runserver` или `migrate`, только нашей: одно слово в
терминале, и готово.

#### Решение

Команды вроде `runserver` и `migrate` — это обычные Python-классы, которые Django
находит и запускает. Свою такую же — её называют **командой управления** — заводят
так же, и тогда она уедет в git вместе с кодом.
Django ищет такие команды в папке `management/commands/` внутри приложения. Создай
структуру (оба `__init__.py` — пустые файлы, по ним Python понимает, что это пакеты):
```
trainer_app/
└── management/
    ├── __init__.py
    └── commands/
        ├── __init__.py
        └── loadquotes.py
```

В файл `loadquotes.py` положи команду — это тот же цикл из 2.9, обёрнутый в класс:
```python
from django.conf import settings
from django.core.management.base import BaseCommand

from trainer_app.models import Quote


class Command(BaseCommand):
    help = 'Загружает цитаты из quotes.txt в базу'

    def handle(self, *args, **options):
        quotes_file = settings.BASE_DIR / 'quotes.txt'
        created = 0
        for line in quotes_file.read_text(encoding='utf-8').splitlines():
            if not line.strip():
                continue
            text, author = line.split('|')
            _, was_created = Quote.objects.get_or_create(text=text, author=author)
            created += was_created
        self.stdout.write(self.style.SUCCESS(f'Готово. Новых цитат: {created}'))
```
Класс наследует `BaseCommand`, а вся работа — в методе `handle`. Сердцевина — тот же
цикл с `get_or_create`, поэтому повтор по-прежнему не плодит дубли (это свойство
называют идемпотентностью). Остальное — непривычные мелочи, и их стоит проговорить.
`get_or_create` возвращает пару «объект, создан ли он только что»; сам объект нам не
нужен, поэтому на его месте стоит `_`, а `was_created` — True или False. В Python
True равен 1, так что, складывая `was_created`, мы считаем новые цитаты.
`*args, **options` — это то, что Django при желании передаёт команде (нам сейчас не
нужно); `self.stdout.write` печатает строку в терминал, а `self.style.SUCCESS` красит
её зелёным.

Запусти команду:
```powershell
python manage.py loadquotes
```
Теперь `loadquotes` стоит рядом с `migrate` и `runserver` — то же, что делал цикл, но
одним словом, повторяемо и из любого места. Запусти ещё раз: новых цитат — 0, дублей
нет.

## 2.10. Главная берёт цитату из базы

Парадокс: данные уже в базе, ими можно управлять из админки и терминала — а главная
страница всё ещё читает старый `quotes.txt` (наш наивный код из 2.1). Про базу она
не знает, и добавленная в админке цитата на сайте не появится.

### Решение

Перепишем view так, чтобы он брал случайную цитату из базы через ORM. Открой
`trainer_app/views.py`:
```python
from django.shortcuts import render

from .models import Quote


def home(request):
    quote = Quote.objects.order_by('?').first()
    return render(request, 'trainer_app/index.html', {'quote': quote})
```
Чтение файла, `random` и `settings` больше не нужны — всё делает база.
`order_by('?')` перемешивает записи в случайном порядке, а `first` берёт первую, то
есть случайную цитату. В шаблон уходит уже целый объект `quote`.

Поправим шаблон `trainer_app/templates/trainer_app/index.html`, чтобы он брал поля
объекта и не падал, если цитат пока нет:
```html
<body>
    {% if quote %}
        <blockquote>{{ quote.text }}</blockquote>
        <p>— {{ quote.author }}</p>
    {% else %}
        <p>Цитат пока нет.</p>
    {% endif %}
</body>
```
`{% if %}` и `{% endif %}` — теги шаблона для условия; `{{ quote.text }}` берёт
атрибут `text` у объекта.

Запусти сервер, обнови главную несколько раз — цитаты меняются, и берутся они теперь
из базы. А вот и ради чего был весь шаг: зайди в админку, добавь новую цитату, обнови
главную — она может тут же на ней появиться. Контентом управляем без правки кода.

> **Честная заметка про `order_by('?')`.** На больших таблицах случайная
> сортировка медленная: базе приходится перемешивать все строки. Для наших
> масштабов это незаметно; когда данных станут миллионы, случайную запись выбирают
> хитрее. Знать про этот предел — часть культуры работы с базой.

## 2.11. Финальный коммит шага

Мы добавили модель и миграции, залили цитаты и переписали главную — но всё это пока
только на диске. Зафиксируем прогресс в машине времени и отправим на GitHub.

### Решение

Попроси Django проверить проект, затем посмотри, что изменилось, подготовь и
проверь короткий список файлов:
```powershell
python manage.py check
git status
git add .
git diff --staged --stat
```
Если в списке только ожидаемое — `models.py`, папка `migrations/`, `admin.py`,
`views.py`, шаблон, `quotes.txt` (и `management/commands/loadquotes.py`, если делал
необязательный подшаг 2.9.1) — коммить и пушь:
```powershell
git commit -m "Добавить модель Quote и случайную цитату из базы"
git push
```

Обрати внимание, чего в списке НЕТ: файла `db.sqlite3`. Он в `.gitignore` — это
локальные данные, а не код, и у каждого они свои. Тогда как же товарищ получит
цитаты? А вот как: `quotes.txt` лежит в git, миграции — тоже. На любом компьютере
достаточно `python manage.py migrate` (создать таблицы) и снова залить цитаты из
`quotes.txt` — циклом в `shell` или, если ты сделал команду из 2.9.1, одним
`python manage.py loadquotes`. Данные воспроизведутся. Это та же идея, что и
`requirements.txt` для окружения: в репозитории не сами данные, а рецепт, как их
получить.

## Новая боль → Шаг 3: связи между данными

Главная показывает цитаты из базы, мы управляем ими тремя способами. Но стоит
присмотреться — и видно новые трещины:

1. **Автор — просто строка.** «Аристотель» повторяется в каждой его цитате. Стоит
   где-то написать «Аристотель » с лишним пробелом — и для базы это уже «другой
   автор». А если захотим хранить про автора что-то ещё (годы жизни, эпоху,
   портрет) — строке это не пришьёшь.

2. **Нет служебных данных.** Мы не знаем, кто и когда добавил цитату. Для сервиса,
   где контент ведут люди, это важно.

3. **Пользователь пока ничей.** Цитаты общие, а наша большая цель — чтобы у каждого
   пользователя была своя страница и своя коллекция.

Решение первой боли — вынести автора в отдельную таблицу и **связать** её с
цитатами (это называется внешним ключом, `ForeignKey`). Со связей между моделями и
начнём Шаг 3, а дальше подойдём к пользователям, их коллекциям и подбору напарников —
к тому самому сервису, ради которого всё затевалось.
